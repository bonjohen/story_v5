# Chapter Assembly: From Scenes to Chapters

*How the ChapterAssembler stitches validated scenes into polished chapter documents. Estimated listening time: 9 minutes.*

---

## The problem chapter assembly solves

The writer agent produces one scene at a time. Each scene is validated against the contract and repaired if needed. But after all scenes are written and validated, you have a collection of loose drafts — not a readable document.

Chapter assembly bridges that gap. It takes the instantiated backbone (which defines the chapter partition) and the validated scene drafts, and produces chapter-level markdown documents with a manifest that maps everything together.

---

## Phase one: Deterministic stitching

The first phase is entirely deterministic — no LLM involved. The assembler walks through the backbone's chapter partition, which groups beats into chapters.

For each chapter, it collects the scenes that belong to the chapter's beats, orders them according to the backbone sequence, and joins them into a single document.

Each chapter starts with a markdown heading — "Chapter 1: Setup" or whatever the partition specified. Between scenes, the assembler inserts a horizontal rule as a scene separator. At the end of the chapter, it appends a metadata footer in an HTML comment: which chapter this is and which scenes it contains. This footer is invisible to readers but essential for traceability.

If a scene draft is missing — perhaps the writer failed on that scene and repair didn't recover it — the assembler inserts a placeholder comment noting the gap rather than silently skipping it.

---

## Phase two: Editorial pass

The second phase is optional and uses an LLM. The editorial agent receives the raw stitched chapter text plus the chapter's editorial constraints — voice, pacing, tone goal, and recap policy.

The editorial agent has a strict mandate: polish prose and fix transitions, but never change structure. It cannot alter plot events, character actions, or the sequence of scenes. It can add brief bridging sentences between scenes, smooth awkward tense shifts, and ensure the narrative voice stays consistent.

The recap policy controls how much context-setting the agent should add. "None" means no recaps — the reader is expected to remember. "Light" means a subtle reference to prior events when transitioning between scenes. "Explicit" means the agent should add a brief summary of relevant prior events at chapter openings.

The editorial agent also preserves metadata footers. If a scene has compliance trace markers — like those generated by the trace engine — the agent must keep them, though it can move them to chapter-level footers.

---

## The chapter manifest

The ChapterManifest is a JSON artifact that describes the complete chapter structure. It lists every chapter with its ID, title, the scene IDs it contains, tone goals, pace directive, editorial constraints, and the file path where the chapter markdown lives.

The manifest also records total counts — total scenes and total chapters — for quick validation. You can check that total_scene_count matches the number of scenes in the plan and that every scene appears in exactly one chapter.

---

## How scenes map to chapters

The mapping flows from the backbone's chapter partition. The partition groups beats into chapters, and each beat contains one or more scenes. So the scene-to-chapter mapping is: scene belongs to beat, beat belongs to chapter.

Within a chapter, scenes appear in backbone order — the order they appear in the beats array. This preserves the narrative sequence that the archetype structure defines.

---

## What the output looks like

A finished chapter document is plain markdown. It starts with a level-one heading, contains scene text separated by horizontal rules, and ends with a metadata comment. The text itself is whatever the writer agent produced (and the editorial agent polished, if enabled).

The manifest sits alongside the chapters as a JSON file. Together, the manifest and the chapter markdown files form a complete, self-describing manuscript artifact.

---

## Design principles

Three principles guide chapter assembly.

First, editorial, not structural. The chapter assembler never rewrites the story's structure. It stitches what the writer produced and optionally smooths the seams. If a chapter's structure is wrong — scenes in the wrong order, wrong beats grouped together — that's a backbone problem, not a chapter assembly problem.

Second, every scene exactly once. The assembler validates that every scene in the plan appears in exactly one chapter. No duplications, no omissions. This is a hard invariant.

Third, metadata preservation. Compliance traces, scene markers, and structural annotations survive chapter assembly. They may be reformatted or relocated to footers, but they're never deleted. This ensures that the final manuscript remains auditable against the original contract.
